**План тестування (Test Plan): Система управління навчанням «Learnyx»**

#### **1\. Вступ (Introduction)**

*   **Мета**: Перевірка працездатності вебсистеми «Learnyx» для автоматизації взаємодії між учнями, викладачами та менеджерами.
    
*   **Об’єкт тестування (In-Scope)**:
    
    *   **Модуль «Authentication & Roles»**: Безпечна авторизація та контроль доступу згідно з ролями (Student, Teacher, Manager).
        
    *   **Модуль «Teacher Availability & Scheduling»**: Ефективне управління слотами викладачів для формування розкладу.
        
    *   **Модуль «Lesson Booking & Management»**: Автоматизація процесу створення та закріплення занять за користувачами.
        
    *   **Модуль «Lesson Lifecycle & Homework»**: Контроль проведення занять, фіксація оцінок та обмін домашніми завданнями.
        
    *   **Модуль «Packages, Cashback & Bonus System»**: Облік пакетів уроків та мотивація учнів через нарахування бонусів.
        
    *   **Модуль «Manager Operations & Dashboards»**: Моніторинг процесів менеджером та надання користувачам актуальної інформації.
        
*   **Поза межами тестування (Out-of-Scope)**:
    
    *   Функціонал сторонніх сервісів відеозв'язку (Google Meet / Zoom), крім збереження посилань у БД.
        
    *   Реальна відправка повідомлень через сторонній Email-провайдер (реалізацію відкладено).
        
    *   Функціонування банківської сторони платіжного шлюзу (LiqPay).
        

#### **2\. Стратегія тестування (Testing Strategy)**

*   **Рівні тестування**:
    
    *   **Integration Testing**: Перевірка коректності передачі даних між React (Frontend) та Django REST API (Backend).
        
    *   **API Testing**: Тестування ендпоінтів через Postman (валідація JWT-авторизації та форматів відповідей).
        
    *   **System Testing**: Наскрізний сценарій (E2E): “Реєстрація учня -> Купівля пакета -> Бронювання уроку -> Проведення -> Нарахування бонусів”.
        
*   **Типи тестування**:
    
    *   **Функціональне**: Перевірка відповідності системи бізнес-вимогам (наприклад, списання уроку з балансу пакета).
        
    *   **UI/UX**: Перевірка наявності індикаторів завантаження та зручності інтерфейсу для різних ролей.
        

#### **3\. Ресурси та інструменти (Tools)**

*   **Інструменти**:
    
    *   **API**: Postman для тестування REST API, Swagger (drf-spectacular) для документації.
        
    *   **UI/Manual**: Браузер Google Chrome (версія 120+) та Chrome DevTools.
        
    *   **Bug Tracking**: Jira для реєстрації дефектів та Google Docs для ведення тест-кейсів.
        
*   **Середовище**: ОС Windows, локальний сервер (Django + React), база даних PostgreSQL.
    

#### **4\. Критерії “Прийняття” (Acceptance Criteria)**

*   **Критерії початку**: Код замерджений у стабільну гілку, база даних розгорнута, наявні тестові облікові записи для всіх ролей.
    
*   **Критерії завершення**:
    
    *   Виконано всі заплановані позитивні та негативні тест-кейси.
        
    *   100% критичних (Critical) та високих (High) багів виправлено та перевірено.
        
    *   Система стабільно працює при одночасному відкритті декількох вкладок під різними ролями.
        

#### **5\. План виконання (Schedule)**

1.  **Етап 1**: Тестування API, авторизації (JWT) та логіки бази даних (тиждень 1).
    
2.  **Етап 2**: Тестування функціональних модулів (бронювання, оплати, домашні завдання) та валідації форм (тиждень 2).
    
3.  **Етап 3**: Фінальне регресійне тестування та перевірка виправлень перед демонстрацією (тиждень 3).
    

#### **6\. Очікувані результати (Deliverables)**

*   **Bug Reports**: Детальна таблиця знайдених дефектів у Jira або Google Таблиці.
    
*   **Підсумковий звіт**: Короткий аналіз функцій, що працюють, та опис наявних обмежень (наприклад, ліміти на розмір файлів домашніх завдань).
